# Design Rationale Ledger: Epic 1.3

## Purpose
This ledger records the requirement-backed architectural decisions that are narrated in `chap-analysis-and-design.tex`.

## Decision Ledger

| Decision | Requirements | Alternatives Considered | Accepted Rationale |
|----------|--------------|-------------------------|--------------------|
| Use event-driven integration with a lightweight orchestrator | FR-012, FR-013, FR-014, FR-020 | Direct REST chain between gateway, ASR, MT, and TTS | Broker-mediated integration isolates scaling domains and preserves diagnosability, while the orchestrator restores sequencing and trace continuity for a logically linear workflow. |
| Use claim-check style object storage for large artifacts | FR-010, FR-011, NFR-007 | Embed large payloads directly in events; rely on presigned URLs internally | Object storage keeps the broker focused on coordination, supports retention and audit, and avoids propagating bearer-style references across internal services. |
| Keep VAD as a separate ingress-stage service | FR-024, NFR-001, NFR-002 | Run ASR directly on raw live stream without pre-filtering | A dedicated VAD boundary removes silence before costly inference and reduces avoidable latency and compute waste. |
| Isolate ASR and TTS as compute-heavy services with Ray Actor option | FR-021, FR-020, NFR-002 | Monolithic model-serving endpoint; per-request model initialization | Warm-loaded Ray Actors reduce initialization cost and keep compute-specific dependencies out of orchestration and gateway services. |
| Separate REST session control from WebSocket streaming | FR-001, FR-004, FR-005 | Single protocol surface for both control and streaming | REST is better suited for session lifecycle and token issuance, while WebSocket supports low-latency audio and transcript exchange. |
| Use Compose for reproducible local runs and Swarm for distributed deployment | FR-018, FR-019, FR-020, FR-022 | Kubernetes-first deployment path | Compose keeps experimentation and smoke testing accessible; Swarm demonstrates replicated deployment and secrets/network handling with lower operational overhead for a thesis scope. |
| Enforce normal-vs-debug telemetry separation | FR-014, NFR-004, NFR-005, NFR-007 | Unrestricted logging in all environments | Metadata-only normal telemetry preserves shareable evidence and privacy, while debug mode remains explicitly opt-in and non-shareable. |
