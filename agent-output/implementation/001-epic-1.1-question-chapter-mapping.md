## Research Question Mapping

| Research Question | Primary Chapter Owner | Supporting Chapters | Architectural or Evidence Anchor | Drafting Rule |
|-------------------|-----------------------|---------------------|----------------------------------|---------------|
| Main question: How can an event-driven microservice architecture be designed and implemented to deliver robust and scalable real-time speech-to-speech translation with measurable performance optimization? | Introduction for framing; Conclusion for final answer synthesis | Analysis and Design, Implementation, Evaluation | Utility tree root, system architecture, evidence ledger | Frame the question in Introduction, justify it through the middle chapters, and answer it only after evaluation evidence is discussed. |
| Sub-question 1: What architectural principles and design patterns support an efficient real-time speech translation pipeline? | Analysis and Design | Background, Implementation | Event-driven pipeline, lightweight orchestrator, schema governance, claim-check, VAD, warm-loaded execution | Use Background for literature context and Analysis and Design for the chosen architectural response. |
| Sub-question 2: What factors affect the system's latency and ability to scale? | Analysis and Design | Evaluation | Latency and scalability drivers, broker decoupling, service-level scaling, artifact persistence, warm vs cold model execution | Separate architectural factors from measured outcomes; the design chapter names the factors, the evaluation chapter measures them. |
| Sub-question 3: How can these factors be measured and optimized in practice? | Evaluation | Implementation | Correlation IDs, trace propagation, benchmark harnesses, quality-evaluation runbook, Normal-vs-Debug telemetry contract | Keep measurement method and optimization narrative tied to reproducible, privacy-safe evidence only. |
| Sub-question 4: How can the system be made robust and available even if individual components fail? | Implementation | Analysis and Design, Evaluation | Retry/backoff, controlled degradation, at-least-once recovery, restart resilience, failure-mode evidence | Describe the mechanisms in Implementation and reserve empirical outcome claims for Evaluation. |
| Sub-question 5: How can the system's performance and quality be evaluated and possibly compared to existing solutions? | Evaluation | Background, Conclusion | Quality runbook, baseline comparison narrative, evidence-ledger deferment discipline | Only make comparison claims that remain active in the evidence ledger; unsupported comparison scope stays deferred. |

## Chapter Ownership Summary

| Chapter | Owned Content |
|---------|---------------|
| Introduction | Problem statement, main question, thesis scope, and contribution framing. |
| Background | Literature context for event-driven systems, speech translation, and comparison framing. |
| Analysis and Design | Architectural drivers, quality attributes, design patterns, service decomposition, and deployment rationale. |
| Implementation | Realized event flow, claim-check handling, warm-loaded execution, VAD guardrails, and robustness mechanisms. |
| Evaluation | Latency, scalability, robustness, quality, and comparison evidence. |
| Conclusion | Final answer to the main question, limitations, and future work. |

## Scope-Lock Notes
- The Evaluation chapter may describe preliminary evidence sources, but definitive empirical claims must follow the activation status in `001-epic-1.1-evidence-ledger.md`.
- Cross-node TLS, production-grade deployment guarantees, and broad external comparisons are not part of the active `v0.1.0` claim set unless later artifacts promote them from deferred to active.
