# Architecture Findings: Epic 1.1 - Thesis Narrative Baseline Setup

## 2. Critical Review & Context

The project aims to deliver a **Real-Time Speech Translation Platform** using an **Event-Driven Microservice Architecture**. The core architectural challenge is the triple-constraint of **Latency**, **Scalability**, and **Robustness** in a distributed AI pipeline (ASR -> MT -> TTS).

### 2.1 Strategic Alignment
The thesis statement (`thesis_statement.tex`) explicitly defines the "Main Question" around event-driven microservice design for robust and scalable translation. The manuscript scaffold (`main.tex` and `chap-analysis-and-design.tex`) matches this by prioritizing:
- Architectural Drivers (Drivers -> Quality Attributes)
- System Decomposition (Microservices)
- Data Modeling (Event schemas)

## 3. Architectural Invariants (The "Golden Rules")
To comply with **Den Gyldne Rengøringsregel** and maintain a defensible thesis, the following invariants MUST be enforced during this baseline setup:

1.  **Traceability Invariant**: Every research sub-question in `thesis_statement.tex` must have a corresponding "Quality Attribute" or "Design Pattern" documented in `chap-analysis-and-design.tex`.
2.  **Evidence Invariant**: All performance claims (latency, throughput) made in the "Evaluation" chapter must be traceable to the raw data files in `agent-output/.../docs/benchmarks/`.
3.  **Boundary Invariant**: The service decomposition must strictly follow event-driven patterns (e.g., Kafka/Ray) to avoid hidden synchronous coupling that would invalidate the "Scalability" narrative.
4.  **Telemetry Baseline**: Normal vs. Debug telemetry must be clearly separated. Normal telemetry (latencies, per-hop) is thesis evidence; Debug telemetry (trace logs) is for implementation verification.

## 4. Risks & Weaknesses
- **Observability Gap**: If the current implementation lack per-hop latency measurement for the event-driven hops, the thesis cannot quantitatively prove the "Sequential data flow optimization" claim.
- **Coupling Risk**: If translation services are synchronously calling each other, the "Robustness" (sub-question 4) cannot be demonstrated via component failure tests.

## 5. Required Telemetry (Draft)
- **Normal**: Correlation IDs across ASR, MT, and TTS; hop-to-hop duration; total end-to-end latency (P50, P95, P99).
- **Debug**: Event schema validation errors; component-level throughput metrics.

## 6. Verdict: APPROVED WITH CHANGES
The baseline setup can proceed provided that the "Mapping Research Questions to Chapters" task in the implementation plan includes a **formal Quality Attribute Utility Tree** mapping sub-questions to architectural drivers.
