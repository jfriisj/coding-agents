# Value Statement and Business Objective

As a **reader and evaluator of the thesis**, I want to **understand the concrete implementation details and how the proposed architecture was realized**, so that **the system's feasibility is proven and the results can be independently reproduced**.

# Objective

The objective of this plan is to draft the complete "Implementation" chapter (`chap-implementation.tex`) for the master's thesis. This chapter will narrate how the high-level design from Epic 1.3 was translated into a functional real-time speech translation platform, focusing on orchestrator logic, data management via the Claim-Check pattern, Ray model actors, and the transition from local development to Docker Swarm orchestration.

# Assumptions

1. The architectural design described in `chap-analysis-and-design.tex` (Epic 1.3) remains the source of truth.
2. The implementation details are based on the existing documentation in `agent-output/docs/` (IaC, Swarm, Runtime, APIS).
3. The "v0.3.0" roadmap release target includes both the Design (Epic 1.3) and Implementation (Epic 1.4) narratives.
4. The LaTeX build system (`Makefile`) is stable and will be used for periodic validation.

# Requirements (Acceptance Criteria)

- **AC1: Orchestrator Logic Documentation**: Detail the event-driven orchestrator, Ray actor lifecycle management, and the `FastAPI` integration. Include the narrative for the **Normal-vs-Debug Observability Implementation**.
- **AC2: Data Management Narrative**: Describe the implementation of the Claim-Check pattern (MinIO/Kafka). Narrate **Short-lived Pre-signed URL Policies** and **SSE-S3 Storage Encryption**.
- **AC3: Model Wrapping and Inference**: Explain the "Warm-Loading" strategy for Whisper and SeamlessM4T models using Ray RServe/Actors and VAD integration.
- **AC4: Reproducibility and Swarm Progression**: Provide a clear path from local `docker-compose` to production `Docker Swarm` deployment. Narrate **Docker Swarm Secrets** usage for sensitive `.env` keys and **Network Overlays** for backend isolation.
- **AC5: Requirement Traceability (Phase 2)**: Link specific implementation details back to the original Functional Requirements (FR) and Non-Functional Requirements (NFR).
- **AC6: Least-Privilege Implementation Narrative**: Document the enforcement of **Kafka Topic ACLs** for service identities (ASR, MT, TTS).

# Plan

## Milestone 1: Orchestration and Event Flow Narrative
1. **Draft Section [sec:impl_orchestrator]**: Describe the Python-based orchestrator and FastAPI integration.
2. **Detail Event Handling**: Explain how Kafka events trigger translation tasks.
3. **Narrate Least-Privilege & Observability**: Detail the implementation of the **Normal-vs-Debug telemetry invariant**. Narrate the use of **Kafka Topic-level ACLs** (Publish/Consume) for service decoupling and security.

## Milestone 2: Data Persistence and Privacy Implementation
1. **Draft Section [sec:impl_data_management]**: Detail the MinIO/Kafka integration.
2. **Explain Claim-Check & URL Lifecycle**: Narrate the use of pre-signed URLs with **Short TTLs (< 1hr)** to minimize bearer token exposure risk.
3. **Security Invariants**: Document the implementation of AES-256 encryption at rest (MinIO SSE-S3) as the "Clean Boundary" for audio data.

## Milestone 3: AI Inference and Ray Integration
1. **Draft Section [sec:impl_model_wrapping]**: Detail the Ray Actor implementation for Whisper/SeamlessM4T.
2. **Warm-Loading Strategy**: Document how models are pre-loaded to minimize cold-start latency (NFR-004 alignment).
3. **Draft Section [sec:impl_guardrails]**: Detail VAD (Voice Activity Detection) integration as an architectural guardrail.

3. **Traceability Mapping**: Ensure every implemented component maps back to a requirement from the specification.
4. **Validation Run**: Perform a full LaTeX build to ensure no broken references between chapters.
