## Detailed Plan

### Milestone 1 (AC1): Requirement-to-Driver Traceability Baseline
- Objective: Derive architectural drivers and quality attributes directly from thesis success criteria and FR/NFR requirements.
- Owner: Implementer
- Dependencies: `requirements-specification.md`, roadmap release framing for v0.3.0, existing requirement-attribute mapping artifact.
- Tasks:
  1. Normalize FR/NFR IDs and map each to one or more quality attributes (latency, scalability, robustness, maintainability, auditability).
  2. Document architecture drivers with rationale tied to thesis success criteria and constraints.
  3. Produce or update `003-epic-1.3-requirement-attribute-mapping.md` as the canonical traceability ledger.
- Acceptance Signals:
  - No requirement relevant to chapter scope is left unmapped.
  - Every listed driver cites requirement-backed rationale.

### Milestone 2 (AC2): Architecture Narrative and Service Decomposition
- Objective: Build the chapter sections that explain high-level architecture, service decomposition, and data governance from implemented artifacts.
- Owner: Implementer
- Dependencies: `docs/component-diagram/`, `docs/sequence-diagram/`, `docs/flow-diagram/`, orchestration and governance runbooks.
- Tasks:
  1. Draft architecture narrative for ingress, orchestrator, ASR, MT, TTS, storage, and event backbone interactions.
  2. Integrate data-governance narrative (artifact lifecycle, retention policy, trace propagation, **Claim-Check/Object Storage boundaries based on ADR 3**).
  3. **Security Narrative Gate**: Define the security posture for data-at-rest (MinIO SSE), data-in-transit (mTLS/WebSocket isolation), and PII scrubbing in the telemetry baseline (ADR 6).
  4. Record justification decisions in `003-epic-1.3-design-rationale-ledger.md`, including the rationale for Kafka vs. REST.
  5. Ensure consistent use of **"Ray Actors"** for inference workers.
- Acceptance Signals:
  - Chapter narrative aligns with existing diagrams and runbooks.
  - Data governance (S3 Claim-Check) and security assumptions (encryption, mTLS) are explicitly stated and internally consistent.
  - Terminology conforms to project standards (e.g., Ray Actors).

### Milestone 3 (AC3): Trade-off Justification and Chapter Integration
- Objective: Demonstrate why event-driven microservices are the selected response to latency, scalability, and robustness trade-offs.
- Owner: Implementer
- Dependencies: Milestones 1-2 outputs, existing benchmark/evidence references, chapter framing from Epic 1.2.
- Tasks:
  1. Draft explicit trade-off argumentation linking architectural choices to latency, scalability, and robustness outcomes.
  2. Consolidate all sections in `chap-analysis-and-design.tex` with requirement and diagram traceability anchors.
  3. Produce `003-epic-1.3-diagram-narrative-sync.md` and `003-epic-1.3-handoff-package.md` for Epic 1.4 continuity.
- Acceptance Signals:
  - Trade-off rationale is explicit, requirement-grounded, and non-contradictory.
  - Chapter is complete and ready for code review and QA/UAT downstream gates.
