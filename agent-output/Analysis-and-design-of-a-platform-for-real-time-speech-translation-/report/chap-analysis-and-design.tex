\section{Architectural Drivers}
\label{sec:arch_drivers}

This chapter converts the design implications from Chapter~\ref{chap:background} into an explicit architectural response for the speech translation platform. The objective is not merely to list components, but to explain why the chosen structure is a defensible answer to the requirements stated in the requirements specification. The primary drivers are therefore derived from the platform's measurable constraints and evidence obligations rather than from technology preference alone. In particular, the architecture must satisfy real-time interaction goals, preserve diagnosability across a distributed pipeline, and remain maintainable enough to evolve as models and deployment targets change \cite{kleppmann2017designing,richardson2018microservices,bass2021softwarearchitectureinpractice}.

\textbf{Latency and interaction quality.} The first driver is end-to-end responsiveness. The platform must deliver first useful output within three seconds and keep segment-level latency within the targets captured by \texttt{NFR-001} and \texttt{NFR-002}. That requirement rules out an architecture that waits for complete utterances before producing value. Instead, the design combines early voice activity detection (\texttt{FR-024}), partial ASR output (\texttt{FR-007}), streaming translation (\texttt{FR-008}), and continuous TTS output (\texttt{FR-009}). Compute-heavy stages are explicitly allowed to run as Ray Actors (\texttt{FR-021}) so that models can stay warm in memory instead of paying repeated cold-start costs.

\textbf{Scalability and change isolation.} The second driver is independent scaling and bounded evolution. Audio ingress, orchestration, ASR, MT, and TTS do not have the same runtime profile, and the requirements explicitly call for asynchronous service-to-service communication (\texttt{FR-012}) and per-service horizontal scaling (\texttt{FR-020}). This pushes the design toward event-driven microservices in which each stage can scale or be replaced without forcing the entire pipeline to be redeployed. Schema governance (\texttt{FR-013}) and contract stability are therefore not secondary implementation concerns; they are architectural enablers that allow the platform to evolve while preserving interoperability.

\textbf{Reliability, auditability, and privacy-safe evidence.} The third driver is trustworthy operation under failure and under evaluation. Retry and controlled degradation (\texttt{FR-015}), trace propagation (\texttt{FR-014}), artifact storage (\texttt{FR-010}), retention control (\texttt{FR-011}), and privacy-safe observability (\texttt{NFR-004} and \texttt{NFR-005}) together require an architecture that can explain what happened without exposing sensitive content. This is why the platform uses correlation and trace metadata as first-class design elements, treats transcripts and audio as sensitive artifacts, and separates shareable normal telemetry from opt-in debug telemetry. The chapter is thus driven by the intersection of performance, operational control, and evidence discipline rather than by throughput alone \cite{bass2023engineeringai,xu2025scalability}.

\section{Quality Attribute Analysis}
\label{sec:quality_attributes}

The quality attributes for the platform are tightly coupled. Improving one dimension, such as modularity, can easily worsen another, such as latency. The design therefore prioritizes a small set of explicit tactics that can be defended against the requirements and against the distributed-systems literature.

\textbf{Performance.} Performance is shaped by the fact that speech translation is a pipeline rather than a single call. The architecture reduces avoidable latency in three ways. First, voice activity detection is used as a gatekeeper so silence and noise do not consume expensive ASR capacity. Second, partial results are propagated through ASR and MT so the UI can display and speak useful output before the user has finished a full utterance. Third, the compute-heavy stages are isolated so warm-loaded Ray Actors can be used where startup overhead is otherwise dominant. The trade-off is that the system accepts more coordination complexity in exchange for lower interaction delay.

\textbf{Scalability and modifiability.} Scalability is not treated as a single global knob. Instead, the design assumes that the gateway, orchestrator, broker, storage, and inference services will scale differently. Event-driven integration allows these stages to be replicated or tuned independently, and the use of schema-managed events keeps those boundaries explicit. This increases operational overhead compared with a monolith or a direct synchronous chain, because messages must be serialized, validated, and traced. The benefit is that the architecture can absorb changes in one stage, for example a different ASR runtime or a new translation model, without rewriting the rest of the pipeline \cite{kleppmann2017designing,newman2019monolith,richardson2018microservices}.

\textbf{Robustness, auditability, and privacy.} Robustness in this platform is inseparable from diagnosability. A service restart, malformed event, or storage outage must be observable through \texttt{correlation\_id}, \texttt{traceparent}, event type, and typed error metadata rather than through raw payload logging. This requirement leads directly to a telemetry policy in which normal mode is always on, low volume, and safe to share, while debug mode is explicit, higher volume, and still forbidden from logging raw audio, transcripts, translations, or credentials. The architecture therefore uses metadata-rich but content-poor telemetry as its baseline evidence posture. That constraint is academically useful because it aligns runtime observability with the thesis requirement for reproducible and privacy-safe evidence.

\textbf{Security as an architectural quality attribute.} Security is addressed as a design constraint rather than a post-processing layer. Session-bound token issuance and the WebSocket AUTH control message protect the ingress path, while internal broker traffic is modeled as an internal-only channel with ACL and mTLS enforcement. For persistent data, MinIO is used with claim-check references and server-side encryption expectations so that large artifacts can be retained for auditability without turning every internal event into a bearer-secret container. The cost is additional governance and configuration complexity, but the resulting design is more defensible because it states where trust boundaries actually exist.

\section{High-Level System Architecture}
\label{sec:high_level_arch}

At a high level, the platform is organized around a thin client-facing ingress layer, an asynchronous event backbone, a small set of bounded AI pipeline services, and persistent stores for state and artifacts. The browser interacts with the system through two ingress surfaces. A REST API is used for session creation, join flow, and token issuance, while a WebSocket endpoint is used for low-latency bidirectional exchange of audio and control messages. This separation keeps short-lived control and long-lived streaming concerns distinct, while still allowing both to participate in the same session context.

Once a session is established, the WebSocket server does not directly call downstream AI services in a synchronous chain. Instead, it emits audio input events onto the broker, where the pipeline is advanced by event consumers and coordinated by a lightweight orchestrator. The broker is the architectural abstraction; Kafka is the chosen implementation. This distinction matters because the chapter's argument concerns decoupled event flow and bounded consumers, not loyalty to one specific product. The orchestrator exists to preserve ordering, correlation, and state progression across a fundamentally asynchronous system, thereby satisfying \texttt{FR-012} without collapsing the design back into direct REST coupling \cite{kleppmann2017designing,hohpe2004enterprise}.

The main processing path follows a clear sequence. The gateway publishes an \emph{AudioInputEvent}. The VAD service filters silence and emits a \emph{SpeechSegmentEvent}. ASR consumes those segments and emits recognized text. MT consumes text and emits translated text. TTS consumes the translated text and emits synthesized audio metadata. The WebSocket server then delivers transcript updates and binary TTS output back to the client. Because these boundaries are explicit, the platform can attribute latency and failure to individual hops instead of treating the whole system as a single opaque inference call.

Large payloads are deliberately kept off the event stream where possible. Audio recordings, segments, transcripts, and synthesized output can be persisted to object storage and referenced through claim-check style handles. This keeps the broker focused on coordination and state progression, while persistent stores carry the heavier audit and retention burden. The result is an architecture in which synchronous control is limited to the user-facing edge, and internal pipeline progress is coordinated asynchronously.

\section{Data Modeling and Governance}
\label{sec:data_modeling}

The platform's core data model consists of four logical units: the session, the segment, the event, and the artifact. A session captures the conversational context, selected languages, and security context. Segments represent bounded processing units derived from the live stream. Events carry lightweight state transitions between services. Artifacts represent the larger persistent outputs, such as raw audio, segmented audio, transcripts, translations, and TTS waveforms. This separation is important because the system has different governance obligations for coordination metadata than for user-derived content.

\textbf{Traceability model.} The architectural spine is \texttt{correlation\_id} plus \texttt{traceparent}. The session identifier provides user-facing continuity, but correlation metadata provides the operational continuity that allows one request path to be followed across gateway, broker, orchestrator, inference services, and storage. This satisfies \texttt{FR-014} and also supports the evidence posture described elsewhere in the project documentation: shareable artifacts should be reconstructible from metadata and deterministic identifiers rather than from raw content dumps.

\textbf{Claim-check storage model.} For auditability, the system stores central pipeline outputs in MinIO with deterministic key structures organized around correlation and segment identity. Internal events carry non-presigned object references rather than bearer-style URLs, which satisfies \texttt{NFR-007}. Presigning, when needed at all, is treated as an edge concern rather than an internal integration pattern. This is a direct application of the claim-check pattern: the broker carries references and coordination metadata, while the object store carries large and potentially sensitive payloads \cite{hohpe2004enterprise}. The documented retention baseline is 30 days, which aligns the storage model with \texttt{FR-011} and with the broader evidence-lifecycle governance used in the project.

\textbf{Privacy-safe telemetry baseline.} Governance also applies to observability data. The normal telemetry mode is intentionally allowlisted to low-volume, stable fields such as \texttt{correlation\_id}, component identity, event type, language metadata, payload sizes, and duration measurements. It explicitly forbids raw audio bytes, transcript text, translated text, query strings, presigned URLs, and credentials. Debug telemetry remains opt-in and non-shareable, but even there the design still avoids raw content and secrets. This distinction is necessary because the same platform must support both runtime diagnosis and thesis evidence packaging without leaking personal data or bearer tokens.

\textbf{Security boundaries.} The governance model also clarifies trust boundaries. Persistent artifacts are expected to use MinIO server-side encryption. Broker traffic is treated as internal-only and protected through ACL and mTLS assumptions. Local development defaults to loopback-bound published ports to avoid accidental LAN exposure. Together these choices make the data path easier to defend academically: the chapter can state where sensitive content is stored, how it is referenced, how long it is retained, and which telemetry classes are safe to share.

\section{Service Decomposition}
\label{sec:service_decomposition}

The platform is decomposed into services according to operational role and scaling behavior rather than according to the internal boundaries of any one model. This decomposition is intended to make the system understandable, replaceable, and measurable.

\begin{itemize}
	\item \textbf{API gateway.} The gateway exposes REST and WebSocket interfaces, terminates the client protocol boundary, validates the initial authentication flow, and translates browser interaction into broker-visible events. It is intentionally thin so user-facing protocol concerns do not leak into inference services.
	\item \textbf{Pipeline orchestrator.} The orchestrator maintains ordering and state progression for the end-to-end workflow. Its role is not to perform inference, but to ensure that asynchronous stages still behave like one coherent pipeline for a given correlation context.
	\item \textbf{VAD service.} VAD is isolated because it acts as a protective efficiency boundary. By filtering silence before ASR, it preserves compute for meaningful speech and reduces noise-induced downstream churn.
	\item \textbf{ASR, MT, and TTS services.} These are the core AI stages. They are separated because they have different runtime dependencies, different scaling profiles, and different opportunities for model substitution. ASR and TTS are especially strong candidates for Ray Actor execution due to warm-model reuse.
	\item \textbf{Broker and schema layer.} The broker provides temporal decoupling and routing, while schema governance protects compatibility between producers and consumers. Together they form the integration contract of the system.
	\item \textbf{Persistent stores.} PostgreSQL stores session-oriented application state, while MinIO stores larger pipeline artifacts. Separating relational state from object artifacts avoids forcing one storage engine to perform both roles poorly.
\end{itemize}

This decomposition is deliberately more granular than a simple API-plus-model monolith, but it is not decomposition for its own sake. Each boundary corresponds to a distinct reason to scale, secure, observe, or evolve that stage differently. The orchestrator and gateway remain comparatively light-weight, whereas the inference services are isolated so they can exploit GPU resources or Ray runtime features without dragging those dependencies into every other component. The resulting design is therefore aligned with maintainability concerns from the literature and with the requirements for independent scaling and diagnosability \cite{newman2019monolith,bass2023engineeringai}.

\section{Infrastructure and Deployment Strategy}
\label{sec:infrastructure}

The deployment strategy follows the same principle as the service decomposition: choose the simplest operational arrangement that still makes the architectural claims testable. Docker Compose is the baseline environment for reproducible local development, smoke testing, and evidence generation. It supports \texttt{FR-018} by making the full stack startable for a new contributor, and it keeps the architecture observable in one place during early validation.

For distributed deployment, Docker Swarm is the release-target topology. This satisfies \texttt{FR-019} and \texttt{FR-020} without introducing the additional operational overhead that a Kubernetes-first strategy would impose on a research project. The choice is therefore pragmatic rather than ideological: Swarm is sufficient to demonstrate replicated services, internal overlay networking, secret distribution, and environment-specific configuration while keeping the deployment argument understandable within the thesis scope. Infrastructure-as-code through Pulumi remains a post-MVP extension path rather than a prerequisite for the architectural core.

Security and evidence handling are also reflected in the deployment model. Internal services such as Kafka, Schema Registry, and MinIO are treated as internal-only surfaces. Local development binds user-facing ports to loopback, and cluster-oriented deployment expects explicit secret handling for storage and broker credentials. This is consistent with the chapter's broader claim that deployment strategy is not separate from architecture: network boundaries, storage trust assumptions, and telemetry policies all shape what the system can safely prove about itself.

Taken together, the Compose-to-Swarm progression supports the value statement of the epic. It shows how the platform can be reasoned about at both development and deployment time, how the microservice boundaries map onto real operational units, and why the event-driven design remains the most coherent response to the combined latency, scalability, robustness, and auditability requirements.
